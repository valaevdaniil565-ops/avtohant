# Функционал проекта и бизнес-логика

## 1. Назначение проекта

`hunting-mvp` — это Telegram-first MVP для рекрутингового/ресурсного мэтчинга.

Система решает 4 прикладные задачи:

1. Собирать вакансии и бенчи специалистов из Telegram, файлов и внешних ссылок.
2. Нормализовать неструктурированные и табличные источники в единый формат.
3. Сохранять вакансии, специалистов, источники и мэтчи в PostgreSQL + `pgvector`.
4. Возвращать менеджеру релевантный TOP кандидатов под вакансию или TOP вакансий под кандидата.

Основной интерфейс работы — менеджерский Telegram-бот. Дополнительно есть collector на Telethon, который слушает Telegram-источники, складывает raw-данные и при необходимости публикует сообщения в архивный канал.

## 2. Главные пользовательские сценарии

### 2.1. Менеджер отправляет вакансию

Бот:

1. Получает текст, файл или внешнюю ссылку.
2. Определяет тип сообщения: вакансия, список вакансий, бенч, список бенчей, нерелевантное.
3. Извлекает одну или несколько вакансий.
4. Сохраняет их в БД.
5. Ищет подходящих специалистов.
6. Применяет бизнес-ранжирование.
7. Возвращает:
   - ручному менеджеру: до `TOP-100` с пагинацией;
   - в ingest-режиме: `TOP-10` и при необходимости рассылает менеджерам.

### 2.2. Менеджер отправляет кандидата / бенч

Поток симметричный:

1. Классификация.
2. Извлечение одной или нескольких карточек специалистов.
3. Сохранение в БД.
4. Поиск подходящих вакансий.
5. Ранжирование.
6. Возврат TOP-выдачи.

### 2.3. Менеджер загружает данные без мэтчинга

В боте есть отдельные режимы:

- `Загрузить вакансии`
- `Загрузить бенч`

В этих режимах сущности сохраняются в БД, но TOP-выдача не строится.

### 2.4. Автоматический ingest из Telegram-источников

Collector:

1. Сканирует доступные каналы/чаты.
2. Делает backfill истории.
3. Сохраняет raw-сообщения в `tg_messages_raw`.
4. Для сообщений из чатов:
   - либо форвардит их в бота;
   - либо публикует в архивный канал, откуда бот уже читает сам.

### 2.5. Синхронизация “нашего бенча”

Есть отдельный реестр собственных специалистов:

1. Ссылка хранится в `app_settings`.
2. Источник регулярно перечитывается.
3. Специалисты сохраняются как `is_internal = TRUE`.
4. Эти записи выделяются в выдаче и участвуют в отдельном блоке `НАШ БЕНЧ`.

### 2.6. Ежедневный digest

Раз в день бот:

1. Обновляет “наш бенч”.
2. Строит digest по новым и обновлённым вакансиям/специалистам.
3. Отправляет digest в заданные чаты.

## 5. Схема данных и бизнес-сущности

### 5.1. `vacancies`

Хранит вакансии:

- роль, стек, грейд;
- опыт, вилку ставки, валюту;
- компанию, локацию, описание;
- `embedding`;
- статус: `active`, `closed`, `hidden`.

### 5.2. `specialists`

Хранит специалистов:

- роль, стек, грейд;
- опыт, ставку, валюту;
- локацию, описание;
- `embedding`;
- статус: `active`, `hired`, `hidden`;
- `is_internal` — собственный специалист или внешний.

### 5.3. `sources`

Трассировка происхождения сущности:

- к какой сущности привязан источник;
- Telegram `channel_id` / `message_id`;
- `message_url`;
- `external_url`, `external_kind`, `external_locator`;
- `source_type`;
- `source_meta`.

Важно: для списков из одного сообщения используется синтетический `message_id = original_message_id * 1000 + idx`, чтобы на каждую сущность была отдельная source-запись.

### 5.4. `matches`

Связка вакансия ↔ специалист:

- `similarity_score`;
- `rank`;
- признак уведомления;
- HR feedback.

### 5.5. `channels`

Реестр отслеживаемых Telegram-источников:

- `source_kind = chat | channel`;
- `is_active`;
- `last_message_id`.

### 5.6. `tg_messages_raw`

Сырой архив Telegram-сообщений collector’а:

- текст;
- entities;
- message URL;
- вложения;
- sender/chat metadata.

### 5.7. `tg_archive_map`

Карта между исходным сообщением и архивным постом:

- origin type;
- classification;
- статус публикации: `pending`, `posted`, `skipped`, `failed`.

### 5.8. `partner_companies`

Справочник компаний-партнёров и счётчик упоминаний.

### 5.9. `app_settings`

Key-value настройки приложения, в том числе:

- ссылка на наш бенч;
- timestamp последнего sync;
- последняя ошибка sync;
- статистика sync.

### 5.10. `own_specialists_registry`

Реестр строк собственного бенча:

- `registry_key`;
- `identity_key`;
- `source_url`, `locator`;
- raw и parsed payload;
- связанный `specialist_id`;
- `parse_status`;
- `is_active`.

## 6. Ключевая бизнес-логика

## 6.1. Контроль доступа

- Бот работает только для `MANAGER_CHAT_IDS`.
- Для ingest-чатов допускается автоматический входящий поток без ручного меню.
- Digest по умолчанию отправляется менеджерам, если отдельные `DIGEST_CHAT_IDS` не заданы.

## 6.2. Классификация сообщений

Используется гибридная схема:

1. Rule-based pre-classifier для коротких bench-card строк.
2. Если уверенности не хватает, вызывается LLM-классификатор.
3. Возможны forced modes из UI:
   - `VACANCY`
   - `BENCH`
   - загрузка без мэтчинга.

Поддерживаемые классы:

- `VACANCY`
- `VACANCY_LIST`
- `BENCH`
- `BENCH_LIST`
- `OTHER`

Дополнительные правила:

- короткие карточки вроде `Java Михаил У. Middle+ 4.8 лет опыта...` определяются rule-based без LLM;
- если LLM вернул `OTHER`, но есть слабый сигнал бенча, может сработать fallback в `BENCH`;
- если в тексте замечено упоминание компании-партнёра и LLM дал `OTHER`, спорный кейс может быть сдвинут в `VACANCY`;
- закрытые/неактуальные сообщения должны уходить в `OTHER`.

## 6.3. Обработка batch/list сообщений

Система умеет распознавать и делить:

- списки специалистов;
- списки вакансий;
- line-wise bench lists;
- табличные документы с несколькими таблицами на одном листе.

Для batch-сообщений preprocessing не должен “схлопывать” границы карточек.

## 6.4. Preprocessing текста перед LLM

Выполняется:

- нормализация Unicode;
- удаление заголовков форвардов;
- чистка markdown-шумов;
- нормализация bullet points;
- мягкое удаление emoji;
- нормализация пробелов;
- извлечение контактов и URL;
- эвристическое “сжатие” длинных текстов, но не для пачек.

## 6.5. Извлечение сущностей

Есть три режима извлечения:

1. `structured_table` — если источник уже распознан как таблица с нормализованными полями;
2. LLM extraction через отдельные prompt’ы для вакансий и специалистов;
3. fallback extraction, если forced mode задан, а LLM ничего не вернул.

## 6.6. Нормализация табличных источников

Из Google Sheets / CSV / XLSX система умеет:

- искать header row не только в первой строке;
- пропускать служебные листы (`Readme`, `Summary`, `Инструкция`, `Архив` и т.п.);
- делить один лист на несколько таблиц;
- определять `entity_hint = BENCH | VACANCY`;
- формировать `structured_fields`;
- проставлять confidence (`high`, `medium`, `low`);
- сохранять sheet/table/index trace.

Если sheet похож на сервисный, он целиком пропускается.

## 6.7. Статусы вакансий и специалистов

### Вакансии

- `active` — доступна для поиска и мэтчинга;
- `closed` — закрытая вакансия;
- `hidden` — скрыта вручную по ссылке.

### Специалисты

- `active` — доступен;
- `hired` — недоступен сейчас;
- `hidden` — скрыт вручную или деактивирован вместе с реестром.

## 6.8. Логика доступности специалиста

Специалист считается недоступным, если:

- в тексте есть сигналы `занят`, `на проекте`, `не ищет`, `unavailable`, `engaged` и т.д.;
- `availability_weeks > 0`.

Специалист считается доступным, если:

- есть сигналы `bench`, `available`, `free`, `на бенче`, `свободен`;
- либо `is_available=True`, и нет явных сигналов недоступности.

## 6.9. Дедупликация сущностей

Для вакансий и специалистов строится `synthetic_id` по:

- роли;
- стеку;
- грейду;
- bucket ставки.

Это даёт idempotent upsert и не позволяет создавать дубликаты при повторной подаче того же материала.

Для собственного бенча дедупликация усилена через `identity_key`, который строится по:

- source URL;
- контактам;
- имени;
- роли;
- стеку;
- при необходимости raw text.

Это позволяет переживать сдвиг строк в таблице без создания нового специалиста.

## 6.10. Мэтчинг и ранжирование

Поиск идёт в два этапа:

1. retrieval:
   - по `pgvector`, если есть embedding;
   - иначе по lexical fallback;
2. business ranking поверх retrieval.

### Обязательный gate

Совпадение по стеку обязательно.

Сначала строится stack profile:

- primary labels;
- secondary tokens;
- tooling labels.

Если primary-overlap нет, кандидат/вакансия отбрасывается, даже если semantic similarity высокая.

### Итоговый business score

Финальный score строится как:

- базовая константа `0.50`;
- `0.15` secondary overlap;
- `0.15` grade score;
- `0.10` rate alignment;
- `0.05` location match;
- `0.05` semantic similarity.

Следствия:

- семантика сама по себе не может победить плохой бизнес-fit;
- слишком дорогой специалист штрафуется;
- для одинакового score выше ставится более дешёвый специалист;
- вакансии с подходящим бюджетом ранжируются выше.

## 6.11. Логика “нашего бенча”

Собственные специалисты:

- хранятся как `is_internal = TRUE`;
- дополнительно связываются с `own_specialists_registry`;
- выводятся в отдельном блоке `НАШ БЕНЧ`;
- в поиске по вакансиям подмешиваются даже поверх обычного retrieval, чтобы не потерялись.

При этом глобальная сортировка всё равно строится по business score, а не по факту “наш/не наш”.

Если в ТОП нет собственных специалистов, бот явно сообщает об этом.

Если sync собственного бенча сломан, в empty-state показывается последняя ошибка и/или время последней успешной синхронизации.

## 6.12. Архивный канал и reference-only посты

Система использует архив в двух разных смыслах:

1. collector публикует туда релевантные сообщения из source-чатов;
2. бот может создавать `reference-only` архив-посты для ручного ввода без исходной Telegram-ссылки.

Reference-only пост создаётся, если:

- это ручной ввод;
- это не ingest из архива;
- это не forward;
- в тексте нет `t.me` ссылки;
- нет внешних URL.

Цель: у каждой сущности должен быть устойчивый traceable source URL.

## 6.13. Удаление / скрытие

Удаление в проекте реализовано как soft hide:

1. менеджер присылает `t.me` ссылку;
2. бот ищет все записи в `sources`, связанные с ней;
3. по подтверждению ставит сущности в `hidden`.

Физического удаления из БД нет.

## 6.14. Sync компаний-партнёров

Если задан `PARTNER_COMPANIES_SOURCE_URL`, система:

1. читает внешний sheet;
2. извлекает компании;
3. учитывает алиасы (`Сбер`, `Sber CIB`, `СберБизнес`, `Лемана Про` и т.п.);
4. накапливает mentions_count;
5. использует этот список как слабый классификационный bias в пользу `VACANCY`.

## 6.15. Ежедневный digest

Digest включает:

- новые вакансии за последние 24 часа;
- обновлённые вакансии;
- новых специалистов;
- обновлённых специалистов.

Скрытые сущности в digest не попадают.

Время запуска — `16:00 MSK`.

## 7. Режимы работы Telegram collector

### 7.1. Backfill

Collector может перечитать историю каналов:

- ограничение по числу сообщений;
- ограничение по окну в днях;
- в backfill сохраняется raw, но не делается форвард/архивный шквал.

### 7.2. Realtime listening

Collector слушает:

- новые сообщения из source-чатов;
- edits;
- ручные посты в архивном канале.

### 7.3. Источники `chat` vs `channel`

Business distinction:

- сообщения из `channel` обычно не архивируются, а сохраняют original route;
- сообщения из `chat` могут архивироваться как bench/vacancy, если прошли лёгкую релевантную классификацию.

### 7.4. Архивный pipeline

Для chat-source сообщения:

1. raw сохраняется;
2. определяется classification: `bench`, `vacancy` или irrelevant;
3. релевантные сообщения публикуются в архив;
4. создаётся запись в `tg_archive_map`.

Если публикация в архив не удалась и включён `archive_fail_open`, сообщение может быть отправлено в бота напрямую.

### 7.5. Вложения

Collector умеет скачивать только часть документов:

- `pdf`
- `doc`
- `docx`

Скачивание ограничено максимальным размером файла и rate limits.

## 8. Поддержка файлов и ссылок

### 8.1. Telegram attachments в боте

Менеджер может прислать:

- CSV
- XLSX
- DOCX
- PDF
- TXT
- HTML

Табличные вложения превращаются в набор ingest units со structured metadata.

### 8.2. External links в тексте

Если в сообщении есть внешняя ссылка:

1. URL извлекается;
2. Telegram-ссылки игнорируются как external source;
3. документ скачивается через MCP fetcher;
4. при успехе каждая строка/карточка превращается в отдельный ingestion unit.

### 8.3. Multi-bench docs

Если документ содержит много коротких line-wise bench карточек, они делятся на отдельные units ещё до LLM extraction.

## 12. Что проект делает по сути

Если коротко, проект автоматизирует операционный цикл рекрутинга/ресурсного менеджмента:

1. собирает рынок вакансий и специалистов из Telegram и документов;
2. приводит разнородные источники к общей модели;
3. отделяет релевантное от шума;
4. сохраняет полный source trace;
5. выделяет собственный bench отдельно от внешнего рынка;
6. строит explainable TOP-выдачу не только по embedding similarity, но и по бизнес-совместимости;
7. поддерживает ручную работу менеджера, автоматический ingest и ежедневный контроль через digest.
